### 1. The JWT Mystery: Why isn't it encrypted?

There is a massive difference between **Encoding** (what a JWT is) and **Encryption** (what people assume a JWT is).

A standard JWT (like the ones generated by your `simplejwt` setup) is like a **postcard**, not a sealed envelope. Anyone who looks at the postcard can read the message on the back, but there is a wax seal on it to prove it came from the king.

**The Anatomy of a JWT:** If you look at a JWT, it has three parts separated by periods: `Header.Payload.Signature`

1. **Header:** Says "I am a JWT and I use the HS256 algorithm."
    
2. **Payload:** This is just Base64-encoded text. If you paste it into a decoder online right now, you will see `{"user_id": 5, "exp": 171000000}`. **It is entirely readable.**
    
3. **Signature:** This is the wax seal.
    

**Why isn't it encrypted (a sealed envelope)?** Because the frontend needs to read the payload! Your Vanilla JS (or a React app) often needs to know "When does my session expire?" or "What is my user ID?" without having to ask the server. If the token was heavily encrypted, the frontend would need the decryption key, which is a massive security risk to store in a browser. _(Note: You should never put sensitive data like passwords or credit cards in a JWT payload for this exact reason)._

**How does decoding and verification actually work?** When the middleware runs `jwt.decode(token, settings.SECRET_KEY)`:

1. It reads the Base64 payload (e.g., `user_id: 5`).
    
2. It takes that payload, combines it with your Django `SECRET_KEY`, and runs it through a one-way math hashing algorithm (HS256).
    
3. It compares the resulting hash to the **Signature** attached to the token.
    
4. If a hacker changed the payload from `user_id: 5` to `user_id: 1` (Admin) and sent it back, Django will run the math, see that the hashes don't match, and reject it. The data isn't hidden, but it is **tamper-proof**.


-------

That is the lightbulb moment! You are 100% correct: **The HMAC (Signature) is different for every single user.**

The backend doesn't verify the token against a "master" token; it verifies the **math** inside that specific token.

Here is how the backend knows "User 3" is actually "User 3":

---

### 1. The Creation (The "Birth" of the Token)

When User 3 logs in, the backend creates the JWT:

- **Payload:** `{"user_id": 3, "exp": 171000000}`
    
- **The Math:** It takes that Payload + your `SECRET_KEY` and creates **Signature_A**.
    
- **The Result:** The backend sends User 3 a token that looks like: `Header.Payload_3.Signature_A`.
    

### 2. The Verification (The "Test")

When User 3 sends that token back later, the backend doesn't look in a database to see "What was User 3's token?" Instead, it does this:

1. It looks at the **Payload** provided in the token (which says `user_id: 3`).
    
2. It takes your `SECRET_KEY` (which only the server knows).
    
3. It runs the HMAC math: `HMAC(Payload_3 + SECRET_KEY)`.
    
4. If the result equals **Signature_A**, the server says: **"The math checks out. I am the one who signed this, and I can confirm this user is ID 3."**
    

### 3. The Hacker's Attempt (The "Forgery")

Now, let's say User 3 is a hacker. They want to be **User 5**.

1. They change the Payload to `{"user_id": 5}`.
    
2. They send the token: `Header.Payload_5.Signature_A`.
    
3. The Backend runs the math: `HMAC(Payload_5 + SECRET_KEY)`.
    
4. **The Result:** This math will produce **Signature_B**.
    
5. **The Catch:** The server compares **Signature_B** (the correct math for User 5) to **Signature_A** (the signature the hacker sent).
    
6. **DENIED:** They don't match. The server knows someone messed with the Payload.
    

---

### The "Secret Sauce"

The reason this works is that the **Signature is a result of the Payload**.

- If the Payload is "User 3", the Signature is "XYZ".
    
- If the Payload is "User 5", the Signature **must** be "123".
    

Since the hacker doesn't have the `SECRET_KEY`, they can't calculate that "123." They are stuck sending the old "XYZ" signature, which only works for "User 3."

### To summarize:

- **Is the HMAC different for each user?** Yes, because the Payload (the user ID) is different.
    
- **Does the backend store all these tokens?** Usually, **no**. That’s the beauty of JWTs! The backend doesn't need a list of active tokens; it just needs the `SECRET_KEY` to "re-calculate" the truth on the fly.
    

Does it make sense now why the `SECRET_KEY` is the most important file in your entire Django project? If a hacker steals that, _then_ they can start forging signatures for any User ID they want!
