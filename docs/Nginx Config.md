**Yes — **most variable names in NGINX are fixed**, and their values come from **the request, the connection, or NGINX itself**. You usually **cannot invent your own variable names** unless you explicitly create them.

Let’s make this clear with examples.

---

# 1. NGINX variables always start with `$`

Anything that looks like:

```nginx
$host
$remote_addr
$uri
$http_upgrade
```

is an **NGINX variable**.

These are predefined by NGINX.

Example from your config:

```nginx
proxy_set_header Host $host;
```

Here:

- `Host` → the HTTP header being sent to Django
- `$host` → a variable containing the domain name

---

# 2. Where the variable values come from

The values come from **three main places**.

---

## 1️⃣ From the client's HTTP request

Example request from browser:

```http
GET /api/users HTTP/1.1
Host: mysite.com
Upgrade: websocket
```

NGINX extracts values from this.

Example variables:

| Variable | Value source |
|------|------|
| `$host` | `Host` header from the request |
| `$uri` | requested path |
| `$http_upgrade` | `Upgrade` header |

Example:

```nginx
proxy_set_header Upgrade $http_upgrade;
```

If the browser sends:

```http
Upgrade: websocket
```

then:

```
$http_upgrade = websocket
```

---

## 2️⃣ From the network connection

Some variables come from the **TCP connection itself**.

Example:

```nginx
$remote_addr
```

This is the **client IP address**.

Example:

```
User IP = 49.36.210.5
```

Then:

```
$remote_addr = 49.36.210.5
```

Used in:

```nginx
proxy_set_header X-Real-IP $remote_addr;
```

---

## 3️⃣ Generated by NGINX internally

Some variables are built by NGINX.

Example:

```nginx
$proxy_add_x_forwarded_for
```

NGINX creates this value by combining:

```
existing X-Forwarded-For header + client IP
```

Example result:

```
203.0.113.1, 49.36.210.5
```

---

# 3. Example request flowing through your config

Browser sends:

```http
GET /ws/chat/ HTTP/1.1
Host: mysite.com
Upgrade: websocket
```

NGINX sees:

```
$host = mysite.com
$uri = /ws/chat/
$http_upgrade = websocket
$remote_addr = 49.36.210.5
```

Then your config sends to Django:

```http
Host: mysite.com
Upgrade: websocket
X-Real-IP: 49.36.210.5
X-Forwarded-For: 49.36.210.5
```

---

# 4. Can you invent new variable names?

Not directly.

This will **NOT work**:

```nginx
proxy_set_header Something $my_variable;
```

because `$my_variable` doesn't exist.

But you **can create variables** using directives like:

### `set`

Example:

```nginx
set $backend_port 8000;
```

Then use it:

```nginx
proxy_pass http://backend:$backend_port;
```

---

# 5. Special variable pattern you saw

One pattern you probably noticed:

```
$http_HEADERNAME
```

Example:

```
$http_upgrade
$http_user_agent
$http_cookie
```

NGINX automatically maps:

```
HTTP header → variable
```

Example:

```
User-Agent → $http_user_agent
Upgrade → $http_upgrade
```

---

# 6. Quick cheat sheet

| Variable | Where value comes from |
|------|------|
| `$host` | Host header |
| `$uri` | requested path |
| `$remote_addr` | client IP |
| `$http_upgrade` | Upgrade header |
| `$proxy_add_x_forwarded_for` | built by NGINX |

---

✅ **Simple rule to remember**

```
$something
```

means:

> "NGINX, insert the value of this request property here."
